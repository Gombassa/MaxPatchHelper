Max MSP AI Assistant

Local RAG assistant for Max 8 / Max for Live → Max 9. Patch creation, guided builds, and object documentation — fully offline.

v0.2 June 2026 Max 8 + M4L → Max 9 Ollama · ChromaDB · Python
1

Goals

Build a fully local AI assistant that understands Max MSP and Max for Live — their objects, signal flow conventions, Live API, and patch structures — and can operate in three distinct modes: guided explanation, direct patch generation, and guided interactive build. The system must serve Max 8 and M4L now, and migrate cleanly to Max 9 without a rebuild.

1.1 Mode Goals

Mode What it does Success looks like
EXPLAIN Answers questions about objects, messages, signal flow, Live API, and Max/M4L conventions in plain language. Cites source pages. Accurate, doc-grounded answers. No hallucinated object names. Reduces time spent in the Max help system and Live API reference.
GENERATE Outputs valid .maxpat JSON for a described Max, MSP, or Max for Live patch or sub-patch. Patches load without errors and behave as described on first or second attempt. M4L patches include correct Live API objects and valid device type declarations.
GUIDED Walks through patch design step by step, confirming object choices and connections before optionally emitting JSON per section. User understands the patch they end up with. Generation is iterative and inspectable. M4L-specific decisions (device type, parameter mapping, API path) are explicitly confirmed.

1.2 Max for Live Scope

Max for Live introduces a distinct layer of requirements beyond standard Max/MSP. The assistant needs to understand and assist with all of the following:

Device types and their constraints

Device TypePrimary useKey constraints the assistant must know
Audio Effect Process audio on a track Requires plugin~ and plugout~. Must handle stereo I/O. No MIDI output by default.
MIDI Effect Process or generate MIDI Uses midiin / midiout. No audio processing. Lives before instruments in the signal chain.
Instrument Synthesise audio from MIDI Receives MIDI, outputs audio via plugout~. Typically uses poly~ for voice management.

Live API — the core M4L layer

The Live API is the main thing that distinguishes M4L development from plain Max. The assistant must be able to explain and generate patches using all of the following API mechanisms:

API ObjectRoleCommon use / gotchas
live.path Define a navigation path in the Live Object Model Paths like live_set tracks 0 clip_slots 0 clip. Must be banged to resolve.
live.object Get/set properties and call functions on LOM objects Works from a resolved live.path id. Properties vs. functions behave differently. Common errors: invalid property names, un-resolved paths.
live.observer Watch a LOM property for changes Set up once; fires on change. Requires path + property name. Must explicitly delete on device unload.
live.remote~ Ramp a LOM parameter at audio rate More efficient than live.object for continuous automation. Needs an id from live.path.
live.dial / live.numbox / live.button / live.slider M4L UI components with parameter binding Expose parameters to Live's automation, Clip envelope, and Push. Parameter name must be unique per device. Range and units set on the object.
live.thisdevice Reference to the device itself in the LOM Essential for self-referencing patches. Outputs id on load and on device move.
live.banks Organise parameters into banks for Push / encoders Optional but expected for Push-compatible devices.

Live Object Model (LOM) awareness

The assistant needs to know the LOM hierarchy well enough to construct valid paths and explain what's accessible where. Key nodes: live_set, tracks, scenes, clips, clip_slots, devices, parameters, master_track, return_tracks.

M4L-specific patterns the assistant should know

PatternDescription
Device initialisation Using live.thisdevice → live.path to resolve the device's own LOM position on load and on move.
Clip launching / querying Reading clip state, launching, stopping, and querying properties like name, length, playing_position via live.observer.
Parameter automation Exposing live.dial/slider parameters and reading back automation values. Understanding the difference between parameter value and parameter name automation.
Transport sync Observing live_set is_playing, current_song_time, and tempo for sync-to-Live behaviour.
Note generation Using live.object to call add_new_notes / set_notes on a clip. The note dict format.
MIDI generation in MIDI Effect Building arpeggiators, sequencers, chord generators that respond to incoming MIDI and generate outgoing MIDI events.
Push 2 / Push 3 integration Parameter bank naming, parameter visibility, and control surface mapping.
Note: M4L patches are still .maxpat files — the format is identical to standard Max patches. What differs is the Live API objects used inside them, the device type declaration, and the UI objects. The generator and validator handle M4L by knowing which objects are M4L-specific and what constraints they carry.

Project Phases

PHASE 1 Scrape, Chunk & Ingest Foundation
Goal
Build the knowledge base from docs.cycling74.com/legacy/max8 and the Live API reference.
Scope
Max 8 object reference pages (all), MSP tutorials, Max topics/guides, Max for Live Live API reference (cycling74.com/legacy/max8 M4L section and docs.cycling74.com/apiref). Jitter excluded.
Scraper
Python + requests + BeautifulSoup. Two crawl targets: main Max 8 docs index, and the Live API reference pages. Strip nav/sidebar, extract main content text.
Metadata tags
Each chunk tagged with: source_url, object_name, doc_type (reference | tutorial | guide | api), domain (max | msp | m4l | live_api), max_version (8).
Chunking
500-token chunks with 50-token overlap. tiktoken for accurate counting.
Embedding
nomic-embed-text via Ollama embedding API. 768-dimension vectors.
Storage
ChromaDB local persistent store. Two collections: max8_docs and max9_docs (empty until Phase 4). Domain metadata enables filtering within a collection.
Inlet/outlet index
Structured JSON mapping object name → inlet list → outlet list. Extracted from reference pages for Max/MSP objects. M4L objects (live.path, live.object, live.observer etc.) added with their I/O indexed separately since they follow different conventions.
LOM reference doc
The Live Object Model hierarchy added as a structured document to the corpus. This is the single most important M4L reference for path construction. Sourced from cycling74.com/legacy/max8/vignettes/live_object_model.
Example patches
10–15 hand-verified .maxpat files covering common patterns: basic audio routing, poly~, pattr, send/receive, metro/bang chains. Plus 5–8 M4L device examples covering one of each device type, a clip launcher, and a transport observer. All stored as annotated description+JSON pairs.
Deliverable
CLI: query the raw vector store and return top-5 chunks for a test question. Confirms ingest worked. Separate test for Max object query and Live API query.
Risk
Some Live API pages (especially LOM property tables) are structured as HTML tables rather than prose. The scraper needs to handle table extraction cleanly or those pages will chunk badly.
PHASE 2 Retrieval Layer & Explain Mode First working assistant
Retrieval pipeline
Embed query → cosine similarity search → return top-6 chunks → inject into prompt. Filter by max_version and optionally domain via ChromaDB metadata. M4L queries auto-filtered to include live_api domain chunks.
Intent classifier
Pre-prompt to the LLM: classify query as EXPLAIN | GENERATE | GUIDED. Also classifies domain: MAX | MSP | M4L. Domain classification determines whether Live API docs are included in retrieval. Single inference call.
M4L detection
Keywords that trigger M4L domain: live.*, M4L, Max for Live, Live API, LOM, device type, ableton, midi effect, audio effect, instrument device. When triggered, LOM reference chunks are always included in context.
Explain prompt
System prompt specifies Max + M4L expertise. For M4L questions: instructs citation of LOM path, property names, and gotchas (e.g. "live.path must be banged to resolve"). Flags uncertainty explicitly.
Model
Mistral 7B for explain mode. Sufficient for prose Q&A with retrieved context.
Validation test set
25 questions: 15 Max/MSP object questions, 10 M4L questions (Live API objects, LOM paths, device type differences, parameter exposure). All must return correct, non-hallucinated answers.
Deliverable
python assistant.py "how do I observe a clip's playing position in M4L?"
PHASE 3 Patch Generation & Guided Build Mode Full dual-mode assistant
Patch schema
.maxpat JSON schema documented as a corpus reference doc. For M4L patches: device type declaration (Audio | Midi | Instrument), UI bounding box conventions for M4L devices, and required Live API boilerplate (live.thisdevice init chain).
Generate prompt
Chain-of-thought: (1) identify device type if M4L, (2) list objects needed, (3) describe each connection, (4) emit JSON. For M4L: step 1 includes confirming which Live API objects are needed and what LOM paths they'll use.
M4L validator
Extended patch validator checks M4L-specific rules: live.thisdevice present if Live API used, device type attribute present, live.path/live.object pairs consistent, live.dial/slider parameter names unique within device.
Guided build — M4L
Guided mode for M4L starts with explicit device type decision before any object selection. Then: what Live API data does the device need? → construct LOM paths → design parameter exposure → design UI. Each step confirmed before proceeding.
Model
Qwen2.5-Coder 14B for GENERATE and GUIDED. Intent classifier routes to this model; Mistral stays for EXPLAIN.
Deliverable (Max)
python assistant.py --mode generate "a patch that plays a sine wave at a set frequency, with a toggle to start and stop"
Deliverable (M4L)
python assistant.py --mode generate "a MIDI effect that arpegiates incoming chords upward over 4 octaves in 16th notes, with a rate dial"
Deliverable (guided)
python assistant.py --mode guided — enters interactive REPL. Guided session for M4L explicitly asks for device type as first step.
PHASE 4 Max 9, Updated M4L & Personal Idiom Layer Production-ready
Max 9 ingest
Re-run scraper against docs.cycling74.com (Max 9 URL). Store in separate ChromaDB collection tagged max_version: 9. Diff against Max 8 collection to surface new/changed objects.
M4L / Live API update
Re-scrape M4L/Live API docs to pick up any Live 12 changes. Diff against existing M4L chunks. Update LOM reference doc if hierarchy changed. Version the LOM doc explicitly.
Version routing
--version flag (default: 8). Routes retrieval to correct collection. Guided build mode surfaces version differences inline if relevant (e.g. "this pattern changed in Max 9").
Personal idioms doc
User-maintained markdown: naming conventions, preferred object choices, common sub-patch patterns, M4L device patterns, LOM path shortcuts, known gotchas. Indexed with retrieval priority above generic docs.
Own patches index
User's existing .maxpat files (both Max and M4L) indexed as examples. Retrieval surfaces them: "you've done something similar in X". M4L device examples labelled by device type.
Deliverable
python assistant.py --version 9 works end-to-end. Personal idiom layer surfaces for test queries about user conventions and own patches.

Architecture

5.1 Request Flow

#ComponentAction
1 CLI entry Parse --mode, --version, --domain flags. If mode omitted, proceed to classifier.
2 Intent + domain classify If mode not specified: Mistral classifies query as EXPLAIN | GENERATE | GUIDED, and domain as MAX | MSP | M4L. Domain classification determines whether Live API chunks are included in retrieval.
3 Structured lookup Check query for known object names. If Max/MSP object: inject inlet/outlet index. If M4L API object: inject m4l_object_index entry. If M4L domain: always inject relevant LOM reference nodes.
4 Retrieval Embed query with nomic-embed-text. Query ChromaDB with version + domain filters. Return top-6 chunks. M4L queries include live_api domain chunks alongside any matching max/msp chunks.
5 Context assembly Combine: structured lookup data + retrieved chunks + conversation history (guided mode only). M4L context always includes LOM hierarchy snippet as preamble.
6 Model dispatch EXPLAIN → Mistral 7B. GENERATE / GUIDED → Qwen2.5-Coder 14B.
7 Response handling EXPLAIN: stream prose. GENERATE: parse JSON, run validator (standard + M4L if applicable), save .maxpat or return error list. GUIDED: stream prose, await confirmation at each stage — for M4L, device type confirmation is stage 0.

5.2 Knowledge Base Layers

PriorityLayerContents
1 — Highest Personal idioms + own patches User's markdown conventions and indexed .maxpat files. Surfaces first if relevant. Own M4L devices labelled by device type for filtered retrieval.
2 Structured indices (not vector-searched) inlet_outlet_index.json, m4l_object_index.json, lom_reference.json. Injected directly on object name / M4L domain detection. Not chunked — deterministic lookup.
3 Example patch pairs Annotated description → .maxpat pairs (Max and M4L). Primary few-shot examples for generate mode. M4L pairs organised by device type.
4 — Baseline Official docs (versioned) Scraped Max 8 / Max 9 reference, tutorials, and Live API reference in ChromaDB. Main knowledge source. Domain metadata enables filtering within a version collection.

5.3 .maxpat JSON Structure

{ "patcher": { "fileversion": 1, "appversion": { "major": 8, "minor": 6, "revision": 5, ... }, "rect": [59, 108, 800, 600], "boxes": [ { "box": { "id": "obj-1", "maxclass": "newobj", // or toggle, number, message, etc. "text": "cycle~ 440", "numinlets": 2, "numoutlets": 1, "patching_rect": [100, 100, 80, 22] // x, y, w, h } } ], "lines": [ { "patchline": { "source": ["obj-1", 0], // [box_id, outlet_index] — zero-based "destination": ["obj-2", 0] // [box_id, inlet_index] — zero-based } } ] } }

5.4 M4L .maxpat Additions

M4L devices are standard .maxpat files with additional required attributes:

{ "patcher": { ... // M4L device type declaration — required "classnamespace": "dsp.devicem4l", // Audio Effect // OR "classnamespace": "mididev", // MIDI Effect // OR "classnamespace": "dsp.devicem4l" + instrument flag "boxes": [ // Required for any device using Live API: { "box": { "id": "obj-1", "maxclass": "newobj", "text": "live.thisdevice", "numinlets": 0, "numoutlets": 2, "patching_rect": [...] } }, // M4L UI objects expose parameters to Live: { "box": { "id": "obj-2", "maxclass": "live.dial", "varname": "Rate", // parameter name (must be unique) "minimum": 0, "maximum": 1, "patching_rect": [...] } }, // live.path for LOM navigation: { "box": { "id": "obj-3", "maxclass": "newobj", "text": "live.path this_device", "numinlets": 1, "numoutlets": 2, "patching_rect": [...] } }, ] } }
Validator M4L checks: device type attribute present · live.thisdevice present if any live.* API objects used · live.path and live.object outlet/inlet pairing consistent · live.dial/slider/numbox parameter names unique within the device · plugin~/plugout~ present for Audio and Instrument devices · midiin/midiout present for MIDI Effect devices.

6.2 Testing Workflow

Explain mode eval (25 questions)

15 Max/MSP questions: object descriptions, message syntax, inlet/outlet behaviour, signal flow. 10 M4L questions: Live API objects, LOM path construction, device type differences, parameter exposure, transport sync patterns. Each answer manually checked against the doc page cited. Pass: correct object names, no invented messages or LOM paths, uncertainty flagged where docs are thin.

Generate mode eval (10 patches)

7 Max/MSP patches of increasing complexity. 3 M4L devices: one per device type. Each opened in Max 8 / Live and tested. Pass: loads without error, behaves as described. Validator must have caught any JSON errors before reaching this stage. M4L devices must expose expected parameters and respond to Live API calls.

Regression

Re-run explain eval after any retrieval or prompt change. Re-run generate eval after any model change or prompt template change.

6.3 Day-to-Day Usage

Explain a Max object

python assistant.py "what does thresh~ do and what are its inlets"

Explain an M4L API object

python assistant.py "how does live.observer work and when do I need to delete it"

Explain an LOM path

python assistant.py "what LOM path do I use to get the tempo of the live set"

Generate a Max patch

python assistant.py --mode generate "a basic sampler: load a buffer, trigger playback with a bang, pitch-shift with a number box"

Generate an M4L device

python assistant.py --mode generate "an M4L MIDI effect that randomly delays incoming notes by 0 to 100ms, with a dial to set the max delay"

Guided build session

python assistant.py --mode guided # For M4L, first prompt: "What type of device? (Audio Effect / MIDI Effect / Instrument)"

Target Max 9

python assistant.py --version 9 "what changed in poly~ between max 8 and 9"
