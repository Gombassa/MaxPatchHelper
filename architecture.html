v1.0 post-review June 2026 9 modules · 1 server · 9 data files

Core Principle

Every file in assistant/ owns exactly one concern. No file duplicates a constant, function, or responsibility owned by another. When you need to add something, the first question is always which file owns this? — not which file is convenient to put it in.

The three rules that follow from this:

Rule 1: Constants defined in config.py are imported everywhere else. Never redefined. Rule 2: Prompt strings live in prompts.py only. Imported by explain.py. Never inline. Rule 3: No file imports from server.py. Server imports from everything; nothing imports from server.

Module Map

Six modules plus the server. Each card shows what the file owns, what it is allowed to import, and its category.

config.py Config
Owns
All constants: paths, URLs, model names, context window sizes. Single source of truth. No logic, no functions.
os nothing from assistant/
prompts.py Prompts
Owns
All system prompt strings. Only EXPLAIN_SYSTEM_PROMPT remains as of the Patch Generator / Guided Spec Builder removal — see §8 Deprecated Features. GENERATE_SYSTEM_PROMPT and GUIDED_SYSTEM_PROMPT (and its sibling learning/spec-extraction prompts) were deleted along with generate.py/guided.py. The MAXCLASS/inlet-outlet reference table that used to live inside GENERATE_SYSTEM_PROMPT was preserved and moved to object_reference.py rather than deleted. No logic, no functions.
nothing — no imports
retrieve.py Retrieval
Owns
ChromaDB queries. Embedding generation via Ollama. Returns raw document chunks to callers. Manages the ChromaDB client (should be singleton — see §7).
config chromadb requests
explain.py LLM · Explain
Owns
Explain mode inference — now the only live LLM flow in the app. Also owns load_inlet_outlet_index(), load_lom_schema(), detect_m4l_context(). These used to be shared utilities imported by generate.py and guided.py in addition to explain.py itself. Both of those files are now deleted (§8), so these are called only internally by explain_query() today. server.py still imports all three by name but no longer calls them directly — a dead-import leftover from the removal, tracked as a new item in §7.2, not yet cleaned up.
config prompts retrieve requests
validate.py Validation
Owns
All patch validation logic. Pydantic models for .maxpat schema. TRUSTED_PORT_COUNTS table. M4L device checks. Returns {valid, errors, warnings}. Never calls Ollama. Confirmed standalone: zero import dependency on generate.py or guided.py, only the reverse (back when they existed). Not wired into any active feature today — reachable only via /api/validate and the test suite — kept intentionally as foundation for a planned future patch-analyzer feature (§8).
config pydantic json nothing else from assistant/
object_reference.py Reference Data
Owns
MAX_OBJECT_REFERENCE_TABLE: MAXCLASS conventions and fixed inlet/outlet counts for ~180 Max objects. Moved out of GENERATE_SYSTEM_PROMPT when generate.py/guided.py were removed (§8), rather than deleted, since it's accurate reference data independent of the generation feature. Not imported by anything yet — prep for the same planned patch-analyzer feature as validate.py. One known error was corrected while moving it: gain~ is 2 inlets/2 outlets (was wrongly listed as 3/2) — see the module docstring for the docs.cycling74.com citation. validate.py's own TRUSTED_PORT_COUNTS["gain~"] = (1, 2) is separately still wrong and remains an open follow-up (§7.2).
nothing — no imports
server.py Entry · API
Owns
FastAPI app. HTTP endpoints only — no WebSocket routes remain (guided's was the only one; removed along with /api/generate in the Patch Generator / Guided Spec Builder removal, §8). CORS pinned to localhost:5173. Routes: /api/health, /api/retrieve, /api/validate, /api/explain (SSE). Imports from explain, validate, retrieve, config. Nothing imports from server.
config explain validate retrieve fastapi
Note on explain.py utility functions: load_inlet_outlet_index(), load_lom_schema(), and detect_m4l_context() used to be shared utilities imported by generate.py and guided.py in addition to explain.py itself. Both of those files are now deleted (§8), so the original "move to a dedicated context.py" concern — updating imports across three files — no longer applies; these functions have exactly one real consumer (explain.py) today. The old suggestion is retired, not carried forward as an open item.

Ownership Rules

When adding or changing anything, match it to the file that owns that concern. If you're unsure, use this table.

If you need to…Touch this fileDo not touch
Add or change a model name, URL, path, or context window size config.py Any other file — import from config instead
Change how the explain prompt is worded prompts.py explain.py — it only imports EXPLAIN_SYSTEM_PROMPT
Add an object to the inlet/outlet reference table object_reference.py validate.py TRUSTED_PORT_COUNTS — keep both in sync if adding a new object. They're already known to diverge on gain~ (§7.2) — don't assume they currently agree before editing either one
Change how patches are validated validate.py Nothing currently calls validate_patch() in production code — only /api/validate and the test suite reach it (generate.py, its only other caller, is deleted)
Change how ChromaDB is queried or embeddings are generated retrieve.py explain.py — it only calls query_vector_db()
Change HTTP routes or CORS server.py Any assistant/ module — they must not know about HTTP. No WebSocket routes remain to worry about (§8)
Change M4L context detection keywords explain.py Nothing external calls detect_m4l_context() directly anymore — it's used only inside explain_query(). server.py still imports it but doesn't call it (dead import, §7.2)

Request Flow

4.1 Explain (Q&A tab)

1 server.py receives POST /explain → calls explain_query() 2 explain.py calls query_vector_db() → retrieve.py returns chunks 3 explain.py calls load_inlet_outlet_index(), detect_m4l_context(), load_lom_schema() 4 explain.py builds user prompt, prepends EXPLAIN_SYSTEM_PROMPT from prompts.py 5 explain.py streams Ollama response → callback → server.py SSE to client └── This is now the only live request flow in the app — Generate and Guided Build flows were removed; see §8 Deprecated Features

Forbidden Patterns

These patterns caused the bugs found in the June 2026 code review. Do not reintroduce them.

PatternWhy it's forbiddenBug ref
Calling requests.post() inside async def without run_in_executor Blocks the entire asyncio event loop. No other requests can be handled while Ollama is thinking. #1
asyncio.create_task() without await in a fire-and-forget pattern Task may never complete on WebSocket disconnect. Silent data loss. #2
Spawning a background thread with no stop event Thread keeps running after WebSocket disconnects, wasting CPU and Ollama resources. #3
TEMPLATE.format(…, query=user_input) where user_input is raw Crashes with KeyError if user types { or } in their query. e.g. "how does {dict} work?" #4
b.maxclass.endswith("~") to detect DSP objects Misses newobj-wrapped audio objects. Use has_object() which checks both maxclass and text fields. #5
chromadb.PersistentClient() inside query functions Creates and tears down the DB client on every query. Use module-level singleton. #6
Opening inlet_outlet_index.json inside per-object validation calls File read repeated once per unknown object in the patch. Cache at module load. #7
allow_origins=["*"] combined with allow_credentials=True Invalid CORS configuration. Browsers reject it and OWASP flags it as a misconfiguration. #8
Defining constants that already exist in config.py Two copies drift apart silently. Import from config instead. #9
Hardcoded relative paths like "data/something.json" Breaks when run from any directory other than the repo root. #10
sys.path.insert() hacks in module files Fragile import system workaround. Use proper package structure with __init__.py instead. #11
Prompt strings defined inline in explain.py or any other assistant/ module Scatters the prompt surface across files. All prompts live in prompts.py only. arch

Deprecated Features

The Patch Generator and Guided Spec Builder features were removed entirely from this codebase. This section is the connective tissue between the old scope (described historically in implementation_plan.md's Phase 3/4 entries, which are not deleted — they're real project history) and the current scope. If anything elsewhere in this document mentions generate.py, guided.py, or classify.py, treat it as historical unless this section says otherwise.

Why: the project is pivoting toward Max 9 and a possible custom MCP server for patch generation, rather than continuing to maintain an in-repo LLM generation/validation loop. Scope has refocused to explain / document search / patch analysis.

Removed backend: assistant/generate.py, assistant/guided.py, assistant/classify.py (deleted entirely — its only purpose was routing between EXPLAIN/GENERATE/GUIDED intents, and with two of three destinations gone, classification was pointless). Routes /api/generate and /api/ws/guided removed from server.py, along with their four WebSocket helper functions. GENERATE_MODEL, GENERATE_CONTEXT_WINDOW, GUIDED_MODEL, GUIDED_CONTEXT_WINDOW removed from config.py. GENERATE_SYSTEM_PROMPT, GUIDED_SYSTEM_PROMPT, GUIDED_LEARNING_SYSTEM_PROMPT, GUIDED_SPEC_EXTRACTION_SYSTEM_PROMPT removed from prompts.py. CLI --mode generate and --mode guided removed from assistant.py; --mode auto removed too, since its only job was routing to generate/guided — an "auto" mode that always resolves to explain isn't auto-detecting anything.

Removed frontend: ui/src/components/Generator.jsx, ui/src/components/GuidedBuilder.jsx. Their tabs, imports, and tab panes removed from App.jsx. connectGuidedWebSocket() and streamGenerate() removed from ui/src/services/api.js — streamPost() and validatePatch() were confirmed shared/intentionally-kept and left alone.

Removed tests: tests/test_generate_guided.py deleted entirely. tests/test_server.py lost test_generate_endpoint_streaming, test_websocket_guided_initialization, test_generate_cancellation. tests/test_validate.py was untouched — confirmed fully independent of generate/guided the whole time.

Kept intentionally, not deleted: assistant/validate.py (standalone, zero dependency on the removed modules — see §2) and the new assistant/object_reference.py (the MAXCLASS/inlet-outlet reference table rescued from GENERATE_SYSTEM_PROMPT rather than deleted with it). Neither is wired into an active feature today; both are foundation for a planned future patch-analyzer feature.

Orphaned/repurposed data: data/personal_idioms.md is orphaned — guided.py was its only reader/writer; the file is left on disk, untouched. data/example_patches/max/ and data/example_patches/m4l/ are repurposed, not orphaned — their fixtures are actively loaded by test_validate.py independent of the deleted generation feature. See §5 for the full breakdown.

For the detailed history of how these features were originally designed and built, see the Phase 3/4 entries in implementation_plan.md — marked deprecated there, not deleted.
